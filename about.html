About

I got into this because a spreadsheet macro wasn't enough.

I started in a small operations team, building internal tools out of necessity — first in Excel, then in Google Apps Script, then in actual JavaScript once I realized the macros were quietly becoming a real application. Nobody told me to learn frontend engineering; I backed into it trying to make a scheduling tool stop breaking every time someone renamed a column.

I studied Computer Science later than most — evenings and weekends, alongside the job that started this — and I still think that order taught me the right lesson: interfaces exist to make someone else's work easier, not to showcase the tool that built them.

Right now, I'm going deeper into motion and interaction design — not because it's trendy, but because I've noticed how much trust a well-timed transition can build in a product, and how rarely that gets treated as engineering rather than decoration. I'm also, honestly, still working on writing documentation as carefully as I write code.

What I optimize for

Clarity over cleverness

A clever abstraction that only I understand is a future bug. I'd rather write the obvious version.

Built for whoever inherits it

I write code and docs assuming someone else — possibly a future, tired version of me — has to maintain this.

Access isn't optional

If it doesn't work with a keyboard or a screen reader, it isn't finished — regardless of how it looks in a demo.
